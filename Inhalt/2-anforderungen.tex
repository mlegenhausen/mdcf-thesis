\chapter{Anforderungsanalyse}
\label{cha:anforderungsanalyse}
In diesem Kapitel werden die sich aus der Aufgabenstellung ergebenen
Anforderungen genauer analysiert. Es wird dabei geklärt, in welchem Umfang die
einzelnen Aspekte bearbeitet werden müssen.

Die für die Arbeit relevanten Anforderungen sind das Sammeln von
Sensordaten, Transparenz, Anonymität, ein Erweiterungssystem, die
Mög\-lich\-keit der Datenspeicherung sowie die regelmäßige Übertragung von Daten
an einen Server.

\section{Sammeln von Sensordaten}
Generell sollen die von den Sensoren anfallenden Daten mit Hilfe der zu
entwickelnden Anwendung gesammelt werden. Über folgende Sensorik verfügen
aktuelle Smartphones:

\begin{itemize}
  \item Mobilfunk (\acs{gsm}, \acs{umts})
  \item \acf{wlan}
  \item Bluetooth
  \item Nahfeldkommunikation (\acs{nfc})
  \item \acf{gps}
  \item Lagesensor
  \item Beschleunigungsmesser
  \item Digitaler Kompass
  \item Näherungssensor
  \item Lichtsensor
  \item Kamera
  \item Mikrofon
\end{itemize}

Die Anwendung soll in der Lage sein, entsprechend auf die Sensorik zuzugreifen
(falls vorhanden) sowie alle anfallenden Daten zu verarbeiten und zu speichern.
Hierbei ist zwischen zwei unterschiedlichen Sensortypen zu unterscheiden: Denjenigen, die potenziell
personenbezogene Daten sammeln, und diejenigen, die keine personenbezogenen
Daten sammeln. Nach §3 des Bundesdatenschutzgesetzes wird der Begriff der personenbezogenen Daten wie folgt
definiert:

\begin{quote}
Personenbezogene Daten sind Einzelangaben über persönliche oder
sachliche Verhältnisse einer bestimmten oder bestimmbaren natürlichen Person.
\end{quote}

Unter Einzelangaben sind z.~B. der Name, die Telefonnummer und die Adresse zu
verstehen, während ``sachliche Verhältnisse'' bedeutet, dass sich über
Daten auf eindeutige Personen schließen lässt. Letzteres ist im Rahmen der
Arbeit von wesentlicher Bedeutung.

Die Sensorik, die potenziell personenbezogene Daten sammelt, sind
Mobilfunk, \acs{gps}, \acs{wlan}, Bluetooth, Kamera und Mikrofon. Die
hierbei anfallenden Daten sind die Position, verfügbare Netzwerke in der
Umgebung, Ton sowie Bildaufnahmen. Über entsprechende Auswertung der Daten
lässt sich auf bestimmte Personen schließen. Diese Daten sind somit besonders
schützenswert. Die zu entwickelnde Anwendung sollte somit über entsprechende
Schutzmechanismen verfügen, die den Zugriff auf solche Daten regelt.

Hierbei ist anzumerken, dass die Anwendung, nach Definition der
Aufgabenstellung, nur in der Lage sein muss, Daten zu sammeln, die von der
Sensorik stammen. Die Ermittlung von Daten wie beispielsweise das
Benutzerverhalten ist nicht zu berücksichtigen.

\section{Transparenz}
\label{sec:transparenz}
Die Forderung nach Transparenz hat im Rahmen der vorliegenden Arbeit das größte
Gewicht. Mit Transparenz ist hierbei nicht gemeint, wie oft in der Informatik,
dass ein bestimmter Prozess ``unsichtbar'' für den Benutzer
abläuft~\cite{transparenz}. Im Rahmen der Erhebung von personenbezogenen Daten
bedeutet Transparenz vielmehr, dass der Benutzer einen genauen Einblick
erhält, welche Daten wie über ihn gesammelt werden. Das bedeutet, dass folgende
Anforderungen an die transparente Datenerhebung gestellt werden:

\begin{itemize}
  \item Der Benutzer erhält zu jedem Zeitpunkt Einblick in die bisher
  gesammelten personenbezogenen Daten.
  \item Der Zugriff auf Systemressourcen, wie z.~B. auf das \acs{gps} wird 
  protokolliert. Dadurch kann ein Benutzer nachvollziehen, auf welche Daten
  zugegriffen und welche Daten gespeichert wurden.
  \item Vor der ersten Datenerhebung kann sich der Benutzer darüber informieren,
  wann welche persönlichen Daten über ihn gesammelt werden.
  \item Eine Verschleierung, also das Erheben persönlicher Daten
  und deren Über\-tragung, ohne die Zustimmung des Benutzers, soll verhindert
  werden.
  \item Der Benutzer erhält Kontrolle darüber, zu welchem Zeitpunkt Daten über
  ihn erhoben werden und wann nicht.
  \item Vor einer Übertragung der Daten an einen externen Server muss der
  Benutzer zustimmen. Sollte er der Übertragung nicht zustimmen, müssen die
  erhobenen Daten verworfen werden.
\end{itemize}

\section{Anonymität}
\label{anonym}
Anonymität beschreibt den Zustand einer Person oder Gruppe, die
nicht identifiziert werden kann. Bei der Art von Anonymität unterscheidet man
zwischen verschiedenen Kategorien~\cite{anonymitaet}:
\begin{itemize}
  \item Formal anonym: Eindeutige Zuordnungsmerkmale (z.~B. Namen usw.) sind
  entfernt, die Daten ansonsten unverändert (d.~h. eine leichte Zuordnung
  der Einheiten).
  \item Faktisch anonym: Die Daten lassen sich nur mit unverhältnismäßigem Zeit-
  und Arbeitsaufwand zuordnen.
  \item Komplett anonym: Die Zuordnung zu einzelnen Personen ist ausgeschlossen.
\end{itemize}

Im Rahmen der Aufgabenstellung soll eine Anonymisierung des Benutzers von
``formal anonym'' bis ``faktisch anonym'' ermöglicht werden. Das bedeutet zum
einen, dass es nicht möglich sein soll, eindeutige Merkmale eines Benutzers zu
sammeln (z.~B. \acs{imsi}, Telefonnummer) und zum anderen, eine Zuordnung zu
realen Personen anhand der Daten zu erschweren.

Beim regelmäßigen übersenden von Datensätzen an den Server treten durch die
Anonymisierung Probleme beim Zuordnen auf. Werden zwei Datensätze A und B zu
zwei unterschiedlichen Zeitpunkten übertragen, ist es nicht möglich, die
Datensätze A und B einander zuzuordnen. Hierbei ergibt sich nun die
Anforderung, ein Verfahren zu entwickeln, dass das Problem löst und
trotzdem die Anonymität des Benutzers wahrt.

\section{Erweiterungssystem}
Wie \acs{boinc} soll die zu entwickelnde Anwendung,
nur ein Grundgerüst darstellen und sich durch Erweiterungen ausbauen
lassen. Die Aufgabe besteht also darin, ein bestehendes Erweiterungssystem
in Android zu integrieren bzw. ein eigenes zu entwickeln.

Anforderungen an das Erweiterungssystem:

\begin{itemize}
  \item Der Benutzer soll die Möglichkeit haben, neue Erweiterungen zu
  installieren und zu deinstallieren.
   \item Die Installation neuer Erweiterungen soll für den Benutzer der
   Anwendung möglichst unproblematisch sein, z.~B. durch die Bereitstellung
   eines {\em Markets}.
  \item Das Erweiterungssystem soll eine Bibliothek liefern, über die es möglich
  ist, neue Erweiterungen möglichst einfach zu erstellen.
\end{itemize}

\section{Datenspeicherung}
Die von den Erweiterungen gesammelten Daten müssen auf dem Gerät
zwischengespeichert werden, bevor eine Übertragung stattfinden darf. Der
Benutzer soll die Möglichkeit besitzen, die gesammelten Daten vor einer
Übertragung zu kontrollieren. Deswegen ist ein Datenmodell zu entwickeln,
welches das Speichern von Daten beliebiger Erweiterungen ermöglicht. Zu beachten
ist hierbei, dass der Entwickler einer Erweiterung so wenig wie möglich bei der
Wahl seines Datenmodells eingeschränkt werden soll.

\section{Datenübertragung}
Die von den Erweiterungen gesammelten Daten sollen in regelmäßigen Ab\-ständen
an einen Server übertragen werden. Dies soll jedoch nicht automatisch geschehen.
Stattdessen soll der Benutzer darauf hingewiesen werden, dass neue Daten zur
Übermittelung bereitstehen, sodass er die Möglichkeit hat, die Daten
vor einer Übertragung zu kontrollieren. Ist ein Benutzer nicht mit der
Übertragung der Daten einverstanden, so soll die Übertragung verworfen werden
können (Siehe \ref{sec:transparenz}). Des Weiteren soll die Möglichkeit bestehen
die Übertragung zu einem späteren Zeitpunkt auszuführen. Das ist dann notwendig,
wenn eine Übertragung z.~B. über das Mobilfunknetz für den Benutzer zu
kostenintensiv ist.

Für die Übertragung der Daten sind ein entsprechendes Datenformat und
Über\-tragungsprotokoll zu wählen. Beim Datenformat ist darauf zu achten, ein
Format zu wählen, das einen geringen Overhead aufweist, da die Bandbreite im
mobilen Bereich stark limitiert ist.

Zum Empfangen der Daten ist eine entsprechende Serverkomponente zu entwickeln.
Neben dem Empfang soll die Serverkomponente es erlauben, die empfangenen Daten
weiter zu verarbeiten, um sie z.~B. in einer Datenbank zu speichern und zu
visualisieren.
