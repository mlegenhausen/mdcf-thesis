\chapter{Anforderungsanalyse}
\label{cha:anforderungsanalyse}
In diesem Kapitel werden die aus der Aufgabenstellung sich ergebenen
Anforderungen genauer analysiert. Es wird dabei geklärt, in welchem Umfang die
einzelnen Anforderungen bearbeitet werden müssen.

Die sich aus der Aufgabenstellung ergeben Anforderungen sind Transparenz,
Anonymität, ein Erweiterungssystem, die Möglichkeit der Datenspeicherung sowie
die regelmäßige Übertragung von Daten an einen Server.

\section{Transparenz}
\label{sec:transparenz}
Das größte Gewicht im Rahmen dieser Masterarbeit hat die Anforderung der
Transparenz. Mit Transparenz wird hierbei nicht, wie oft in der Informatik
gemeint, dass ein bestimmter Prozess ``unsichtbar'' für den Benutzer
abläuft~\cite{transparenz}. Im Rahmen der Erhebung von personenbezogenen Daten
bedeutet Transparenz, dass der Benutzer einen tiefen Einblick darin erhält,
welche Daten, wie über ihn gesammelt werden. Dies bedeutet, dass folgende
Anforderungen an die transparente Datenerhebung gestellt werden:

\begin{itemize}
  \item Der Benutzer erhält zu jedem Zeitpunkt, Einblick in die bisher
  gesammelten personenbezogenen Daten.
  \item Vor der ersten Datenerhebung kann sich der Benutzer darüber informieren,
  welche persönlichen Daten, wann über ihn gesammelt werden.
  \item Eine Verschleierung, also das erheben von persönlichen Daten
  und deren Übertragung, ohne die Zustimmung des Benutzers soll verhindert
  werden.
  \item Der Benutzer soll Kontrolle darüber haben, wann Daten über ihn erhoben
  werden sollen und wann nicht.
  \item Vor einer Übertragung der Daten, an einen externen Server, muss der
  Benutzer zustimmen. Sollte er der Übertragung nicht zustimmen, sollen die
  erhobenen Daten verworfen werden.
\end{itemize}

\section{Anonymität}
\label{anonym}
Anonymität beschreibt den Zustand einer Person oder Gruppe, die
nicht identifiziert werden kann. Bei der Art von Anonymität unterscheidet man
verschiedene Abstufungen~\cite{anonymitaet}:
\begin{itemize}
  \item Formal anonym: Eindeutige Zuordnungsmerkmale (z.~B. Namen usw.) sind
  entfernt, die Daten ansonsten unverändert (das heißt eine leichte Zuordnung
  der Einheiten).
  \item Faktisch anonym: Die Daten lassen sich nur mit unverhältnismäßigem Zeit-
  und Arbeitsaufwand zuordnen.
  \item Komplett anonym: Die Zuordnung zu einzelnen Personen ist ausgeschlossen.
\end{itemize}

Im Rahmen der Aufgabenstellung soll eine Anonymisierung des Benutzers
von ``formal anonym'' bis ``faktisch anonym'' ermöglicht werden. Dies bedeutet,
dass es nicht möglich sein soll, eindeutige Merkmale eines Benutzers zu sammeln
(z.~B. \acs{imsi}, Telefonnummer) und zum anderen, eine Zuordnung zu realen
Personen anhand der Daten erschwert wird.

Bei der Anonymisierung der Daten ergibt sich aber ein Problem, bei der
Zuordnung von einzelnen Datensätzen, die in regelmäßigen Abständen an einen
Server übertragen werden. Werden zwei Datensätze A und B zu zwei
unterschiedlichen Zeitpunkten übertragen, ist es nicht möglich, die
Datensätze A und B einander zuzuordnen. Hierbei ergibt sich nun die
Anforderung, ein Verfahren zu entwickeln, dass dieses Problem löst und die
trotzdem die Anonymität des Benutzers bewahrt.

\section{Erweiterungssystem}
Aus der Aufgabenstellung ist zu entnehmen, dass die zu entwickelnde Anwendung,
nur ein Grundgerüst darstellen soll. Es soll sich durch Erweiterungen erweitern
lassen. Die Anforderung besteht also darin, ein bestehendes Erweiterungssystem
in Android zu integrieren, oder ein eigenes zu entwickeln.

Anforderungen an das Erweiterungssystem:

\begin{itemize}
  \item Der Benutzer soll die Möglichkeit haben neue Erweiterungen zu
  installieren und zu deinstallieren.
   \item Die Installation neuer Erweiterungen soll für den Benutzer der
   Anwendung möglichst einfach sein z.~B. durch die Bereitstellung eines
   {\em Markets}.
  \item Das Erweiterungssystem soll eine Bibliothek liefern, über die es möglich
  ist, neue Erweiterungen möglichst einfach zu erstellen.
\end{itemize}

\section{Datenspeicherung}
Die von den Erweiterungen gesammelten Daten müssen auf dem Gerät
zwischengespeichert werden, bevor diese übertragen werden dürfen. Der Benutzer
soll nämlich die Möglichkeit besitzen, sich die gesammelten Daten, vor einer
Übertragung zu kontrollieren. Deswegen ist ein Datenmodell zu entwickeln, dass
es ermöglicht, Daten von beliebigen Erweiterungen zu speichern. Der Entwickler
einer Erweiterung soll dabei aber so wenig wie möglich bei der Wahl seines
Datenmodells eingeschränkt werden.

\section{Datenübertragung}
Die von den Erweiterungen gesammelten Daten sollen in regelmäßigen Ab-ständen,
an einen Server übertragen werden. Dies soll aber nicht automatisch geschehen,
sondern es soll der Benutzer darauf hingewiesen werden, das neue Daten zur
Übermittelung bereitstehen. Dadurch soll der Benutzer die Möglichkeit haben,
vor einer Übertragung die Daten zu kontrollieren. Ist ein Benutzer nicht mit der
Übertragung der Daten einverstanden, so soll diese Übertragung verworfen werden
können (Siehe \ref{sec:transparenz}). Des Weiteren soll die Möglichkeit bestehen
die Übertragung zu einem späteren Zeitpunkt auszuführen. Dies ist dann nützlich,
wenn eine Übertragung z.~B. über das Mobilfunknetz für den Benutzer zu
kostenintensiv ist.

Für die Übertragung der Daten ist ein entsprechendes Datenformat und
Über-tragungsprotokoll zu wählen. Beim Datenformat ist darauf zu achten, ein
Format zu wählen, das einen geringen Overhead aufweist, da die Bandbreite im
mobilen Bereich stark limitiert ist.

Zum Empfangen der Daten ist eine entsprechende Serverkomponente zu entwickeln.
Neben dem Empfang soll die Serverkomponente es erlauben, die empfangenen Daten
weiter zu verarbeiten, um sie z.~B. in einer Datenbank zu speichern und zu
visualisieren.
